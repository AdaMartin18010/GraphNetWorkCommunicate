## 📊 **二、概念多维矩阵 / Multi-dimensional Concept Matrices**

### 2.1 系统模型对比矩阵

| 维度 | 同步模型 | 异步模型 | 部分同步模型 |
|------|---------|---------|------------|
| **时间假设** | 有上界 | 无上界 | 部分有界 |
| **消息延迟** | 有界 | 无界 | 动态有界 |
| **时钟** | 同步 | 不同步 | 部分同步 |
| **适用场景** | 局域网 | 互联网 | 混合网络 |
| **算法复杂度** | 简单 | 复杂 | 中等 |
| **容错能力** | 强 | 弱 | 中等 |
| **实际应用** | 实时系统 | 大规模系统 | 云系统 |

### 2.2 一致性模型对比矩阵

| 维度 | 强一致性 | 最终一致性 | 因果一致性 | 会话一致性 |
|------|---------|-----------|-----------|-----------|
| **定义** | 所有节点立即看到相同数据 | 系统最终达到一致状态 | 保持因果关系的一致性 | 同一会话内保持一致性 |
| **关系** | 最强一致性级别 | 弱一致性的一种 | 介于强一致和最终一致之间 | 基于会话的一致性 |
| **延迟** | 高 | 低 | 中等 | 低 |
| **可用性** | 低 | 高 | 中等 | 高 |
| **适用场景** | 金融系统、交易系统 | 社交网络、DNS | 分布式存储、消息队列 | Web应用、缓存系统 |
| **CAP权衡** | 选择CP | 选择AP | 平衡 | 选择AP |

## 🌲 **四、证明树图 / Proof Trees**

### 4.1 CAP定理证明树

```mermaid
graph TD
    Theorem[CAP定理<br/>不能同时满足C、A、P三个性质] --> Assume[假设: 系统同时满足<br/>C(一致性)、A(可用性)、P(分区容错性)]

    Assume --> Partition[场景: 网络分区发生<br/>系统被分为两部分]

    Partition --> Case1[情况1: 保证一致性C<br/>必须等待分区恢复<br/>拒绝写操作]

    Partition --> Case2[情况2: 保证可用性A<br/>必须接受写操作<br/>允许不同分区独立操作]

    Case1 --> Conflict1[矛盾: 违反可用性A<br/>无法响应请求]
    Case2 --> Conflict2[矛盾: 违反一致性C<br/>不同分区数据不一致]

    Conflict1 --> Contradiction[矛盾: 无法同时满足<br/>C、A、P三个性质]
    Conflict2 --> Contradiction

    Contradiction --> Conclusion[结论: CAP定理成立<br/>最多只能同时满足两个性质]

    style Theorem fill:#e1f5ff
    style Conclusion fill:#d4edda
    style Contradiction fill:#fff3cd
```

### 4.2 FLP不可能性定理证明树

```mermaid
graph TD
    Theorem[FLP不可能性定理<br/>异步系统中无法保证确定性共识] --> Assume[假设: 存在确定性共识算法<br/>在异步系统中保证共识]

    Assume --> Async[异步系统特性:<br/>消息延迟无界<br/>无全局时钟]

    Async --> Scenario[场景: 只有一个节点可能失效<br/>其他节点正常]

    Scenario --> Delay[消息延迟:<br/>可以任意延迟消息<br/>但最终会到达]

    Delay --> Indistinguishable[不可区分性:<br/>故障节点和延迟节点<br/>无法区分]

    Indistinguishable --> Contradiction[矛盾: 算法无法判断<br/>节点是故障还是延迟<br/>导致无法达成共识]

    Contradiction --> Conclusion[结论: 异步系统中<br/>无法保证确定性共识]

    style Theorem fill:#e1f5ff
    style Conclusion fill:#d4edda
    style Contradiction fill:#fff3cd
```

## 🔄 **五、控制执行数据流图 / Control Flow & Data Flow Diagrams**

### 5.1 分布式系统通信流程

```mermaid
flowchart TD
    Start([开始分布式通信]) --> Send[节点A发送消息:<br/>创建消息m<br/>设置目标节点B]

    Send --> Route[路由消息:<br/>通过网络路由<br/>传递到节点B]

    Route --> Receive[节点B接收消息:<br/>接收消息m<br/>验证消息完整性]

    Receive --> Process[处理消息:<br/>执行相应操作<br/>更新本地状态]

    Process --> Response{需要<br/>响应?}

    Response -->|是| CreateResp[创建响应消息:<br/>生成响应r<br/>设置目标节点A]

    Response -->|否| Complete[通信完成]

    CreateResp --> Route2[路由响应:<br/>通过网络路由<br/>传递到节点A]

    Route2 --> ReceiveResp[节点A接收响应:<br/>接收响应r<br/>处理响应]

    ReceiveResp --> Complete
    Complete --> End([分布式通信结束])

    style Start fill:#e1f5ff
    style End fill:#d4edda
    style Process fill:#fff3cd
```

### 5.2 故障检测与恢复流程

```mermaid
flowchart TD
    Start([开始故障检测]) --> Heartbeat[发送心跳:<br/>节点定期发送<br/>心跳消息]

    Heartbeat --> Wait[等待响应:<br/>等待心跳响应<br/>设置超时时间]

    Wait --> Check{收到<br/>响应?}

    Check -->|是| Normal[节点正常:<br/>更新最后心跳时间<br/>继续监控]

    Check -->|否| Timeout[超时:<br/>心跳超时<br/>未收到响应]

    Timeout --> Suspect[怀疑故障:<br/>标记节点为怀疑状态<br/>增加超时次数]

    Suspect --> Verify{确认<br/>故障?}

    Verify -->|否| Heartbeat
    Verify -->|是| Detect[检测到故障:<br/>确认节点故障<br/>触发恢复流程]

    Detect --> Recovery[故障恢复:<br/>选择恢复策略<br/>执行恢复操作]

    Recovery --> Replicate[数据复制:<br/>从备份节点<br/>恢复数据]

    Replicate --> Replace[节点替换:<br/>启动新节点<br/>替换故障节点]

    Replace --> End([故障恢复结束])
    Normal --> Heartbeat

    style Start fill:#e1f5ff
    style End fill:#d4edda
    style Detect fill:#fff3cd
```

### 5.3 CAP权衡决策流程

```mermaid
flowchart TD
    Start([开始CAP权衡决策]) --> Analyze[分析业务需求:<br/>一致性要求<br/>可用性要求<br/>分区容错性要求]

    Analyze --> Priority{确定<br/>优先级?}

    Priority -->|一致性优先| CheckAvail{可用性<br/>要求?}

    Priority -->|可用性优先| CheckConsist{一致性<br/>要求?}

    CheckAvail -->|高| Conflict1[矛盾:<br/>无法同时满足<br/>需要权衡]
    CheckAvail -->|低| CP[选择CP系统:<br/>强一致性<br/>牺牲可用性]

    CheckConsist -->|高| Conflict2[矛盾:<br/>无法同时满足<br/>需要权衡]
    CheckConsist -->|低| AP[选择AP系统:<br/>高可用性<br/>牺牲一致性]

    Conflict1 --> Choose1[根据业务场景<br/>选择主要目标]
    Conflict2 --> Choose2[根据业务场景<br/>选择主要目标]

    Choose1 --> CP
    Choose2 --> AP

    CP --> ImplementCP[实现CP系统:<br/>强一致性协议<br/>如2PC、Paxos]

    ImplementCP --> End([CAP权衡决策结束])
    ImplementAP --> End

    style Start fill:#e1f5ff
    style End fill:#d4edda
    style Priority fill:#fff3cd
```
